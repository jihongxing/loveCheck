# 公司举报功能实施文档

## 📋 功能概述

在原有的恋爱信用预警平台基础上，新增**公司举报功能**，允许用户举报存在劳动违法行为的公司，帮助求职者规避"黑心企业"。

---

## 🎯 核心功能

### 1. 公司举报 (Company Report)
- **举报主体**: 公司名称 + 工商注册号（可选）
- **举报内容**:
  - 行业分类
  - 所在城市
  - 就职时间段
  - 职位（可选，脱敏）
  - 违法行为标签（拖欠工资、强制加班、PUA管理等）
  - 详细描述（最多5000字）
  - 证据附件（劳动合同、工资条、聊天记录等，最多9个文件）

### 2. 公司查询 (Company Query)
- **查询方式**: 公司名称 + 工商注册号
- **查询结果**:
  - 举报总数
  - 风险评分（0-100）
  - 风险等级（低/中/高）
  - 违法行为标签聚合
  - 涉及城市
  - 时间跨度
  - 详细举报记录列表

### 3. 公司申诉 (Company Appeal)
- 公司可提交申诉说明
- 上传反证材料
- 公开陪审团投票机制

### 4. 陪审团投票 (Jury Vote)
- 用户可对举报/申诉双方投票
- IP + 设备指纹防刷
- 每日投票上限（20票）
- 达到阈值自动清洗记录

## 🗄️ 数据库设计

### 新增表结构

#### 1. `company_records` - 公司举报记录表
```sql
CREATE TABLE company_records (
    id SERIAL PRIMARY KEY,
    company_hash CHAR(64) NOT NULL,              -- 公司名+注册号的HMAC-SHA256哈希
    company_name VARCHAR(200) NOT NULL,          -- 完整公司名
    display_name VARCHAR(200),                   -- 脱敏公司名（如：北京字**科技有限公司）
    registration_no VARCHAR(100),                -- 脱敏注册号（如：9111****234567）
    industry VARCHAR(100),                       -- 行业分类
    location_city VARCHAR(200),                  -- 公司所在城市
    description TEXT,                            -- 详细描述
    risk_level INT DEFAULT 1,                    -- 风险等级 1=低 2=中 3=高
    tags JSONB DEFAULT '[]',                     -- 违法行为标签数组
    evidence_mask_url TEXT,                      -- 证据文件URL数组（JSON）
    status VARCHAR(30) DEFAULT 'active',         -- active/hidden/cleansed_by_jury
    reporter_hash CHAR(64) NOT NULL,             -- 举报人身份哈希
    reporter_display_name VARCHAR(80),           -- 举报人匿名昵称
    verification_level INT DEFAULT 1,            -- 验证等级 1=基础 2=有合同 3=有法律文书
    reporter_city VARCHAR(100),                  -- 举报人所在城市
    employment_period VARCHAR(100),              -- 就职时间段（如：2023.01-2024.06）
    position VARCHAR(100),                       -- 职位（可选）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_hash_company USING HASH (company_hash),
    INDEX idx_active_company_hash (company_hash) WHERE status = 'active',
    INDEX idx_company_records_tags USING GIN (tags)
);
```

#### 2. `company_stats` - 公司申诉与投票统计表
```sql
CREATE TABLE company_stats (
    company_hash CHAR(64) PRIMARY KEY,
    appeal_reason TEXT,                          -- 申诉理由
    appeal_evidence TEXT,                        -- 申诉证据URL数组（JSON）
    reporter_votes INT DEFAULT 0,                -- 支持举报人的票数
    company_votes INT DEFAULT 0,                 -- 支持公司申诉的票数
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_company_stats_hash USING HASH (company_hash)
);
```

#### 3. `company_watchlists` - 公司关注列表（预留）
```sql
CREATE TABLE company_watchlists (
    id SERIAL PRIMARY KEY,
    company_hash CHAR(64) NOT NULL,
    user_hash CHAR(64) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_company_watchlist_company USING HASH (company_hash),
    INDEX idx_company_watchlist_user USING HASH (user_hash)
);
```

## 🔧 后端API设计

### 1. 提交公司举报
```
POST /api/v1/company/report
Content-Type: multipart/form-data

参数:
- reporter_phone: 举报人手机号（必填，后台实名，前台匿名）
- company_name: 公司名称（必填）
- registration_no: 工商注册号（可选，提高匹配精度）
- industry: 行业分类（可选）
- location_city: 公司所在城市（可选）
- employment_period: 就职时间段（可选）
- position: 职位（可选）
- tags: 违法行为标签JSON数组（必填）
- description: 详细描述（必填，最多5000字）
- evidence_files[]: 证据文件（可选，最多9个，每个最大10MB）

响应:
{
  "message": "company_report_submitted",
  "record_id": 123
}
```

```
GET /api/v1/company/query?company_name=xxx&registration_no=xxx
```

### 3. 公司申诉
```
POST /api/v1/company/appeal
Content-Type: multipart/form-data

参数:
- contact_phone: 联系人手机号（必填）
- company_name: 公司名称（必填）
- registration_no: 工商注册号（可选）
- reason: 申诉理由（必填，最多5000字）
```

## 🔒 隐私与安全设计

### 1. 零知识查询
- 公司名 + 注册号通过 `HMAC-SHA256` 加盐哈希存储
- 数据库仅存储哈希值，无法反向推导
- 查询时实时计算哈希进行匹配

### 2. 举报人保护
- **后台实名**: 手机号哈希存储，防止恶意举报
- **前台匿名**: 显示随机昵称（如：举报者#A7B3），不暴露真实身份
- 地理位置仅显示城市级别

### 3. 防刷机制
- **Bloom Filter**: 快速过滤不存在的公司，减少数据库查询
- **Redis限流**: 
  - 写操作：10次/分钟
  - 读操作：60次/分钟
  - 投票：20次/天（IP + 设备指纹双重防护）
- **投票去重**: 
  - IP地址去重（30天TTL）
  - 设备指纹去重（30天TTL）

### 4. 证据存储
- MinIO对象存储，独立于数据库
- 文件名前缀标识：`comp_` (举报证据) / `comp_apl_` (申诉证据)
- 访问需要HMAC签名Token（24小时有效期）
- 支持图片和PDF格式
- 文件魔数验证，防止Content-Type伪造

## 📊 风险评分算法

### 评分维度
1. **举报数量权重**: 
   - 1-2条: 基础分 20
   - 3-5条: 基础分 40
   - 6-10条: 基础分 60
   - 10+条: 基础分 80

2. **标签严重性权重**:
   - 高危标签（拖欠工资、性骚扰、违法裁员）: +15分/条
   - 中危标签（强制加班、PUA管理）: +10分/条
   - 低危标签（其他）: +5分/条

3. **证据完整性加成**:
   - 有证据附件: +10分
   - 有劳动合同/法律文书: +20分

4. **陪审团投票调整**:
   - 如果公司申诉且获得支持，评分按比例降低
   - 公式: `最终分 = 原始分 × (举报人票数 / 总票数)`

### 风险等级划分
- **低风险**: 0-40分
- **中风险**: 41-70分
- **高风险**: 71-100分
