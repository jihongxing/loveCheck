为了快速验证（MVP）这个想法，我们需要一套**极轻量、低成本、高隐蔽性**的技术方案。既然你具备系统架构背景，我们可以跳过复杂的 K8s 分布式设计，直接采用“**Serverless 思想 + 嵌入式数据库**”的结构，在一周内上线跑通逻辑。

以下是针对“快速验证”设计的技术实现路径：

---

## 1. 核心技术栈 (The Lean Stack)

- **前端：** **Uni-app / Vue3 (H5)**。适配微信内打开，开发速度最快。
    
- **后端：** **Go (Gin 框架)**。利用你熟悉的 Go 语言，单文件部署，内存占用极低。
    
- **数据库：** **SQLite**。无需安装数据库服务，直接是一个本地文件，符合你“本地资源管理”的需求。
    
- **存储与缓存：** 证据图片统一采用 **MinIO** 对象存储。必须引入 **Redis** 负责高频数据的缓存和 API 接口的高级分布式限流防刷。
    

---

## 2. 零知识查询逻辑（防审计核心）

为了确保即使数据库被拖库也无法还原信息，采用 **“Salted Hash”** 机制：

- **入库：**
    
    - 用户输入手机号 `13812345678`。
        
    - 后端生成 `Query_Hash = Bcrypt(13812345678 + Global_Salt)`（使用慢速哈希防止彩虹表暴破）。
        
    - 数据库**仅存**这个 Hash 字符串和脱敏后的显示名（如：张*某）。
        
- **查询：**
    
    - 查询者输入完整手机号。
        
    - 后端实时计算 Hash 并与数据库比对。
        
    - _结论：_ 只有持有完整号码的人才能查到结果，任何人都无法通过数据库反推号码。

## 3. 极简数据库设计 (SQLite Schema)

SQL

```
-- 风险记录表
CREATE TABLE risk_records (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    target_hash TEXT NOT NULL,          -- 手机号/身份证的加盐Hash
    display_name TEXT,                  -- 脱敏姓名，如 "张*某"
    location_city TEXT,                 -- 发生城市
    risk_level INTEGER DEFAULT 1,       -- 风险等级 (1-低, 2-中, 3-高)
    tags TEXT,                          -- 行为标签，JSON存储: ["冷暴力", "经济纠纷"]
    evidence_mask_url TEXT,             -- 模糊化后的证据图路径
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 支付意向表 (验证付费意愿)
CREATE TABLE payment_intents (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    record_id INTEGER,
    user_ip TEXT,
    status TEXT DEFAULT 'pending'       -- pending, success
);
```

## 4. 快速验证的“障眼法”支付方案

在验证阶段，**不要**去申请复杂的微信支付接口（审核极难通过）：

- **方案 A：** 点击“查看详情”时，弹出打赏二维码（个人收钱码），备注“查询+ID”。
    
- **方案 B（推荐）：** 采用“**付费咨询制**”。页面显示“由于隐私保护，详细证据需联系人工风控官核实”，点击跳转到企业微信或个人微信号。
    
    - _目的：_ 这种方式不仅能收钱，还能通过沟通深入了解用户的真实痛点，作为后续迭代的输入。

## 5. 部署方案：一台服务器搞定

既然你熟悉 Docker，直接写一个 `docker-compose.yml`：

1. **Caddy/Nginx：** 负责 HTTPS 证书和 H5 静态资源托管。
    
2. **Go-App：** 后端 API。
    
3. **SQLite File：** 挂载在宿主机目录。
    
4. **Redis & MinIO：** 容器化部署用于限流缓存和独立图片资源存储。
    

**上线流程：**

1. **域名购买：** 买一个看起来像“信用查询”或“风险预警”的域名（如 `love-risk-check.com`）。
    
2. **内容录入：** 预先从公开信息、社交平台搬运 10-20 条已公开的典型案例作为基础数据，防止用户查不到任何东西而流失。
    
3. **流量测试：** 在小红书、豆瓣的相关情感话题下发帖，引导用户搜索。

## 6. 开发者提醒：你不可忽视的“逃生门”

在代码实现中，请务必预留一个 **“一键销毁”** 接口或简单的管理面板：

- **反爬早与反虚假诬陷（防女巫攻击）：** 采用**“前台匿名、后台实名”**机制（用户提供微信认证或手机验证以形成后台哈希身份），结合 **Redis** 实现精准的滑动窗口限流（IP+设备指纹拦截高频机器请求）。
    
- **内容管理：** 如果收到律师函或投诉，确保你能一键把 `risk_records` 中的某条记录状态改为 `hidden`。
